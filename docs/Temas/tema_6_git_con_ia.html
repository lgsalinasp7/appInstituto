// tema 05
Git con IA
Criterio técnico sobre las sugerencias — la IA es una herramienta, no el piloto.

La IA puede sugerirte comandos que destruyen el historial de un repositorio compartido con total confianza y sin ninguna advertencia.

Un git push --force origin main sugerido sin que tú entiendas las consecuencias puede borrar el trabajo de todo el equipo de KaledSoft de una semana.

El criterio técnico es tuyo. La IA es solo una herramienta.

// 01 — concepto base
La IA como par de programación
No como reemplazo del juicio. No como el experto final.
✓ en qué sí ayuda la IA
Tareas donde brilla
— Entender mensajes de error confusos de merge
— Generar borradores de mensajes de commit
— Explicar qué hace un diff grande
— Sugerir el comando para una acción específica
— Traducir errores crípticos de Git al español
⚠ en qué falla la IA
Tareas donde debes tener cuidado
— No conoce el estado actual de tu repositorio
— No sabe si hay compañeros trabajando en la misma rama
— Puede sugerir soluciones que funcionan en su contexto imaginado pero destruyen el tuyo
— Presenta comandos destructivos con el mismo tono que los seguros
💡 el modelo mental correcto
La IA es un junior muy listo
Imagina que tienes un compañero nuevo en KaledSoft que sabe mucho de Git en teoría, pero no conoce el proyecto, no sabe qué ramas están en producción, ni quién está trabajando en qué.

Le puedes pedir sugerencias. Pero tú eres quien decide si ejecutar el comando o no. Tú eres el senior. La IA es el junior.
flujo recomendado con IA
1. Describe el problema a la IA
2. Lee el comando sugerido
3. Entiéndelo antes de ejecutarlo
4. Verifica que aplica a tu caso
5. Ejecuta con criterio
// 02 — zona de peligro
Comandos peligrosos que la IA sugiere
Estos tres pueden destruir trabajo. Nunca los ejecutes sin entender exactamente qué hacen.
!
git push --force origin main destruye historial remoto
Sobreescribe el historial del repositorio remoto con tu versión local. Si hay 5 personas en KaledSoft que hicieron commits esta semana y tú tienes una versión vieja localmente, con este comando borras el trabajo de todos. Git no pregunta. No hay confirmación. No hay papelera de reciclaje.
!
git reset --hard HEAD~5 elimina 5 commits locales
Elimina los últimos 5 commits de tu historial local sin posibilidad de recuperación simple. Todo el código que escribiste en esos commits desaparece del árbol de trabajo. La IA suele sugerir esto para "limpiar" el historial, sin aclarar que es irreversible si no tienes un reflog activo.
!
git clean -fd borra archivos no rastreados
Elimina permanentemente todos los archivos que Git no está rastreando. Si tienes archivos nuevos que creaste pero aún no hiciste git add, este comando los borra del disco para siempre. No van a la papelera. Desaparecen.
🔑 regla de oro
Si un comando sugerido por la IA contiene --force, --hard, o -f, detente. Busca qué hace exactamente. Pregúntale a la IA que te explique las consecuencias antes de ejecutarlo.
// 03 — problemas frecuentes
Los 6 problemas más comunes que genera la IA con Git
Patrones reales que aparecen cuando usas IA para resolver problemas de Git sin supervisión.
⚠ problema 1
Contexto imaginado
La IA no tiene acceso a tu repositorio. Asume un estado que no existe. Puede sugerirte un merge estrategia basada en una rama que en tu repo tiene un nombre diferente o ya fue eliminada.
⚠ problema 2
Confianza uniforme
La IA presenta git status y git push --force con el mismo tono tranquilo y profesional. No hay señales de alerta. Tú eres quien debe reconocer la diferencia.
⚠ problema 3
Soluciones de Stack Overflow antiguas
La IA fue entrenada con código y respuestas de internet que pueden tener años. Puede sugerir workflows de Git obsoletos que el equipo de KaledSoft ya no usa.
⚠ problema 4
Conflictos de merge mal resueltos
Si le pegas un diff a la IA y le pides que resuelva el conflicto, puede elegir la versión equivocada sin entender la lógica de negocio de KaledDental. La IA elige sintácticamente, no semánticamente.
⚠ problema 5
Commits demasiado grandes
La IA suele sugerir mensajes de commit genéricos que agrupan demasiados cambios. "fix: multiple issues" no le dice nada a nadie en una revisión de código seis meses después.
⚠ problema 6
Historial reescrito en rama compartida
Si la IA sugiere git rebase o git commit --amend en una rama que ya está en el remoto y que otros usan, estás reescribiendo historial compartido. Catástrofe garantizada.
// 04 — supervisión
El checklist antes de ejecutar cualquier comando sugerido por IA
5 preguntas que deben responderse antes de presionar Enter.
1
¿Entiendo exactamente qué hace este comando? Si no puedes explicarlo en una oración, no lo ejecutes. Pídele a la IA que te lo explique paso a paso primero.
2
¿Este comando modifica el historial remoto? Cualquier cosa que toque el repositorio remoto afecta a todos en KaledSoft. Busca si hay origin, remote, o --force en el comando.
3
¿Hay compañeros trabajando en esta rama ahora mismo? Revisa con tu equipo antes de hacer operaciones destructivas en ramas compartidas. Un mensaje en Slack puede salvarte.
4
¿Tengo un punto de regreso seguro? Antes de cualquier operación riesgosa, crea una rama de respaldo: git branch backup-$(date +%Y%m%d). Costo: 2 segundos. Valor: tu trabajo.
5
¿Puedo probarlo en seco primero? Muchos comandos de Git tienen la opción --dry-run o -n que te muestra qué haría sin hacerlo. Úsala siempre que exista.
// 05 — uso correcto
Cómo pedirle bien a la IA sobre Git
La calidad de lo que obtienes depende de cómo preguntas. Aquí la diferencia.
caso de uso ❌ prompt malo ✓ prompt bueno
Error de merge "cómo soluciono este error de git" "Tengo este error de merge en Git en la rama feature/kaled-dental-billing. Explícame qué significa cada parte del conflicto antes de darme la solución"
Mensaje de commit "genera un mensaje de commit" "Estos son los cambios que hice: [pega el diff]. Genera 3 opciones de mensaje de commit siguiendo Conventional Commits. Explica por qué elegiste cada verbo"
Revertir cambios "cómo revierto mis cambios" "Quiero deshacer mi último commit pero SIN eliminar los archivos del disco. ¿Qué comando uso? ¿Hay riesgo si ya hice push de esa rama?"
Revisar un diff "revisa este código" "Este es el diff de un PR en KaledWash. Explícame qué cambió, si hay algo potencialmente peligroso, y si los cambios son coherentes con el mensaje del commit"
Resolver conflicto "soluciona este conflicto" "Tengo este conflicto de merge. La función en HEAD calcula precios con IVA, la de la rama nueva los calcula sin IVA. ¿Cuál debo conservar y por qué? No me des el código aún."
💡 patrón general
Dale contexto → pide explicación antes que solución → especifica las restricciones → valida antes de ejecutar.
// 06 — recuperación
Cómo revertir errores — el kit de supervivencia
Cuando algo sale mal, Git casi siempre tiene una salida. Aquí las principales.
🔧 herramienta 1
git revert — la forma segura
git revert abc1234 seguro en ramas compartidas
Crea un nuevo commit que deshace los cambios del commit indicado. No reescribe el historial. Perfecto para ramas que ya están en el remoto y que otros usan.
🔧 herramienta 2
git reflog — la máquina del tiempo
git reflog
# muestra TODO lo que hiciste
git reset --hard HEAD@{3}
Git guarda un log de todos tus movimientos por ~90 días. Incluso si hiciste reset --hard, el reflog puede rescatarte. Es la red de seguridad oculta de Git.
🔧 herramienta 3
git reset --soft — volver sin perder código
git reset --soft HEAD~1 código intacto
Deshace el último commit pero deja todos los archivos modificados en el staging area. Es como "descomprometer" sin perder el trabajo. Diferente a --hard que borra todo.
🔧 herramienta 4
git stash — guardar y salir limpio
git stash temporal
git stash pop # recuperar
git stash list # ver todos
Guarda cambios no committeados en un cajón temporal. Útil cuando necesitas cambiar de rama urgentemente en KaledSoft sin commitear trabajo incompleto.
// 07 — arqueología de código
git blame — entender el pasado
Cuando encuentras un bug en KaledDental, git blame te dice quién escribió esa línea y en qué contexto.
⬡ KaledDental — archivo: src/billing/calculateTax.ts
$ git blame src/billing/calculateTax.ts
a3f9c21 carlos.m 2024-03-12 1 export function calculateTax(amount: number) {
a3f9c21 carlos.m 2024-03-12 2 const TAX_RATE = 0.19;
f2b8e55 sofia.r 2024-07-08 3 const TAX_RATE = 0.08; // ← bug introducido aquí
a3f9c21 carlos.m 2024-03-12 4 return amount * TAX_RATE;
a3f9c21 carlos.m 2024-03-12 5 }
💡 lo que git blame te dice
— El commit exacto que introdujo esa línea (f2b8e55)
— Quién hizo el cambio (sofia.r)
— Cuándo fue (2024-07-08)
— Luego puedes ver el PR completo con git show f2b8e55
✓ no es para culpar
El nombre del comando es desafortunado. git blame no es para culpar a nadie — es para entender el contexto histórico. Sofia pudo haber cambiado el rate por un requerimiento del cliente que luego cambió de nuevo. El historial cuenta la historia completa.
// 08 — mensajes de commit
Generar mensajes de commit con IA — bien hecho
La IA puede ayudarte a escribir mensajes claros. Pero tú validas que sean precisos.
❌ mensajes malos (IA sin guía)
git commit -m "fix"
git commit -m "changes"
git commit -m "update code"
git commit -m "fix: multiple issues"
No dicen qué se arregló, qué cambió, ni por qué. En 6 meses nadie en KaledSoft podrá entender el historial del proyecto.
✓ mensajes buenos (Conventional Commits)
feat(billing): add IVA calculation for KaledDental

fix(auth): resolve token expiry on KaledWash login

refactor(parking): extract slot availability logic

docs(api): update endpoint docs for KaledPark
Formato: tipo(scope): descripción corta. Claro, buscable, y genera changelogs automáticamente.
💬 prompt para pedirle a la IA un buen mensaje de commit
# Copia el diff de tus cambios y usa este prompt:

"Estos son mis cambios en Git: [diff]. Genera 3 mensajes de commit
siguiendo Conventional Commits. El proyecto se llama KaledDental.
Explica brevemente por qué elegiste el tipo para cada uno."
Así la IA te da opciones y te explica su razonamiento. Tú eliges la más precisa para lo que realmente hiciste.

✓ úsala para
entender errores generar commit msgs revisar diffs explicar comandos resolver merge (con supervisión) encontrar el comando correcto

1

2
El reflog es tu red de seguridad git reflog guarda ~90 días de historial. Casi siempre hay regreso.
3
git revert es el camino seguro Para deshacer en ramas compartidas, siempre revert sobre reset.
4
El criterio es tuyo La IA es rápida. Tú eres preciso. Juntos son poderosos. Separados, peligrosos.

En KaledSoft, el historial de Git es la memoria del equipo. La IA puede ayudarte a escribirla más rápido — pero tú eres el guardián de que sea correcta.
