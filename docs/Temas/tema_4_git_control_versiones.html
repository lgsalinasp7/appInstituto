Antes de Git · El caos colaborativo

Así trabajaban los equipos
antes de Git

😱 El método carpetas

Cada cambio = una nueva carpeta. kaledwash_v1, kaledwash_v2, kaledwash_PRODUCCION_FINAL. Cuando hay 3 devs trabajando esto explota en minutos.

📧 El método email/FTP

Dev A manda archivos por email. Dev B sube por FTP. Dev C no sabe qué versión tiene. Alguien sobreescribe el trabajo de otro. Se cae producción. Todos se culpan.

🗄️ CVS (1990) y SVN (2000)

CVS fue el primer intento serio: un servidor central guardaba todos los cambios. SVN (Subversion) lo mejoró: historial, comparaciones, ramas. Pero eran centralizados — sin internet, no podías trabajar. Y hacer ramas era lento y doloroso.

💡 El problema fundamental

¿Cómo saber exactamente qué cambió, cuándo, y quién lo cambió? ¿Cómo trabajar en paralelo sin destruir el trabajo ajeno? ¿Cómo volver atrás si algo explota en producción?

KaledSoft sin Git: Ana trabaja en KaledDental, sube por FTP. Carlos modifica el mismo archivo para KaledWash. Overwrite. El sistema de citas de KaledDental deja de funcionar. Nadie sabe por qué. 😬
Historia · 2005

Linus Torvalds y el
nacimiento de Git

1991

Linux nace

Linus crea el kernel de Linux. Miles de devs del mundo empiezan a contribuir.

2002

BitKeeper (gratis para Linux)

El equipo de Linux usa BitKeeper, una herramienta propietaria que les dan gratis. Es distribuido y rápido.

2005

BitKeeper retira la licencia gratis

Un dev de Linux intentó hacer ingeniería reversa de BitKeeper. La empresa revoca la licencia. Crisis total.

2005

🎉 Git nace en 10 días

Linus, furioso, decide construir su propio sistema. En 10 días crea el primer prototipo funcional de Git. El 7 de abril de 2005 hace el primer commit.

2008

GitHub lanza

GitHub convierte Git en social. Cualquiera puede hostear y colaborar. Cambia la industria para siempre.

¿Por qué Git ganó todo?

  • ▸ Distribuido — cada dev tiene el historial completo. Sin internet, sigues trabajando.
  • ▸ Ramas baratas — crear una rama tarda milisegundos, no minutos.
  • ▸ Inmutable — cada commit tiene un hash único (SHA-1). No se puede alterar el historial sin que se note.
  • ▸ Gratis y open source — para siempre.

Primeros pasos · Configuración

Configurar Git por
primera vez

Antes de hacer cualquier commit, Git necesita saber quién eres. Esto se guarda en el archivo ~/.gitconfig.

terminal — configuración inicial
$ git config --global user.name "Ana García"
$ git config --global user.email "ana@kaledsoft.com"
# Verificar configuración guardada
$ git config --list
user.name=Ana García
user.email=ana@kaledsoft.com
# Configurar editor de texto
$ git config --global core.editor "code --wait"
# Inicializar un repositorio nuevo
$ cd kaledsoft-app

/kaledsoft-app/.git/

El corazón de Git · Commits

El commit: la fotografía
del sistema

Un commit es una instantánea completa de todos tus archivos en ese momento, identificada por un hash SHA-1 único e irrepetible.

Anatomía: Conventional Commits

feat ( kaledwash ): agrega turnos WhatsApp
↑
tipo
↑
scope
↑
descripción
# Tipos de commit más comunes
feat → nueva funcionalidad
fix → corrección de bug
docs → cambios en documentación
refactor → mejora sin nueva feature
chore → tareas de mantenimiento
$ git commit -m "feat(kaledental): agrega recordatorio SMS citas"
$ git commit -m "fix(kaledpark): corrige cálculo tarifa nocturna"
$ git commit -m "hotfix(kaledwash): máquina 3 no registraba ciclo"
git log --oneline — KaledSoft
a3f9c12 feat(kaledwash): agrega turnos WhatsApp
8e2b401 fix(kaledental): bug en agenda sábado
c71da3e feat(kaledpark): módulo reservas anticipadas
19f3b80 docs: actualiza README instalación
4a9c210 feat(kaledwash): integración pasarela Wompi
# Ver diff de un commit específico
$ git show 8e2b401
# Ver diferencias sin commitear
$ git diff
# Corregir el último commit
$ git commit --amend

¿Por qué conventional commits?

  • ✓ Genera changelogs automáticos
  • ✓ Cualquier dev entiende el historial
  • ✓ Permite versioning semántico automático
  • ✓ Filtra por producto: git log --grep="kaledwash"

# Crear y cambiar a una nueva rama
$ git checkout -b feature/kaledwash-pagos-wompi
# O con el comando moderno
$ git switch -c feature/kaledwash-pagos-wompi
# Ver todas las ramas
$ git branch
  develop
* feature/kaledwash-pagos-wompi
  main
# Cambiar de rama
$ git switch develop
# Unir rama feature → develop
$ git merge feature/kaledwash-pagos-wompi

🔒 Regla #1: Nunca commit directo a main

main = lo que está corriendo ahora mismo para los pacientes de KaledDental.
Todo cambio llega via Pull Request con revisión de al menos 1 compañero. Si necesitas un fix urgente: rama hotfix/ → PR → merge.

Comandos de ramas esenciales

git branch -d feature/xxx ← borrar rama local
git stash ← guardar cambios temporalmente
git stash pop ← recuperar cambios guardados
git log --graph ← árbol visual de ramas

El flujo completo de Ana

  1. Crea feature/kaledwash-pagos
  2. Hace commits mientras desarrolla
  3. Abre Pull Request en GitHub → develop
  4. Carlos revisa y aprueba
  5. Merge a develop, se borra la feature branch
  6. Semanal: develop → main con tag de versión
Seguridad · .gitignore

Lo que NUNCA debe
subir al repositorio

⚠️ El escenario de pesadilla

Carlos sube .env a GitHub público por accidente. Ahí están las credenciales de Neon (base de datos de KaledDental). En 15 minutos los bots de GitHub ya las encontraron. Alguien tiene acceso a todos los historiales médicos. Esto pasa en el mundo real todo el tiempo.

# Dependencias — son reproducibles
node_modules/
# Build de Next.js — se regenera
.next/
out/
# Sistema operativo
.DS_Store

# Editor de código
.vscode/settings.json
Archivo / Carpeta Qué contiene en KaledSoft Por qué ignorar
.env DB_URL de Neon, CLERK_SECRET, WOMPI_KEY Credenciales privadas
node_modules/ Miles de paquetes npm (~500MB) Se instala con npm install
.next/ Build compilado de Next.js Se genera en cada deploy
*.log Logs de servidor, errores Cambián constantemente, innecesario
.DS_Store Metadata interna de macOS Basura del sistema operativo

✅ Solución: .env.example

Sube .env.example con los nombres de las variables (sin valores). Cada dev crea su propio .env con sus credenciales reales.

DATABASE_URL=tu_url_de_neon_aqui
CLERK_SECRET=tu_clave_clerk_aqui
WOMPI_KEY=tu_key_de_wompi_aqui
💡 Pro tip: gitignore.io genera .gitignore automáticos para cualquier stack

$ git log -p historial.js
# ¿Quién escribió esta línea? (blame)
$ git blame historial.js

$ git checkout 8e2b401 -- historial.js
# Revertir un commit (crea nuevo commit inverso)
$ git revert a3f9c12
# Traer un commit específico de otra rama
$ git cherry-pick a3f9c12
git pull vs fetch

fetch = descarga cambios sin aplicar
pull = fetch + merge automático

⚠️ git reset --hard

Descarta commits. Peligroso en ramas compartidas. Preferir revert.

🔀 Pull Request (PR) — el corazón de la colaboración

  1. Ana sube su rama feature/kw-pagos
  2. Abre PR en GitHub: "feature/kw-pagos → develop"
  3. GitHub muestra exactamente qué líneas cambiaron
  4. Carlos revisa, comenta, sugiere cambios
  5. Ana corrige, hace nuevos commits al mismo PR
  6. Carlos aprueba ✅ → merge automático
  7. GitHub borra la rama feature automáticamente

Funcionalidades clave de GitHub

  • ▸ Issues — rastrear bugs y tareas de KaledSoft
  • ▸ Actions — CI/CD: tests automáticos en cada PR
  • ▸ Protección de ramas — bloquear push directo a main
  • ▸ Code review — comentar líneas específicas
  • ▸ Releases y tags — marcar versiones de producción
KaledSoft config: main protegida → requiere 1 aprobación + tests passing antes de merge.
