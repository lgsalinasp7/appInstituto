El problema de trabajar en equipo

Imagina que 3 developers de KaledSoft trabajan al mismo tiempo en el mismo archivo

😱 Sin ramas
🧑‍💻

Ana arregla un bug en el formulario de citas de KaledDental

🧑‍💻

Carlos agrega el pago con QR para KaledPark

🧑‍💻

María actualiza el diseño del dashboard de KaledWash

💥

Los 3 hacen commit a main al mismo tiempo. El código de Carlos sobreescribe el trabajo de Ana. María no sabe qué pasó. Producción se cae a las 3 PM.

✅ Con ramas
🌿

Ana trabaja en su propia rama: fix/kaledental-citas

🌿

Carlos en su rama: feature/kaledpark-qr

🌿

María en su rama: feature/kaledwash-dashboard

🎯

Cada una trabaja independiente. Cuando terminan, revisan el código y lo unen a main de forma ordenada. Nadie pisa el trabajo de nadie.

¿Qué es una rama?

Una rama es una copia paralela del código donde puedes experimentar sin afectar el original

🏠

main

El código oficial. Lo que ven los clientes reales.

🌿

Tu rama

Tu espacio seguro para trabajar sin miedo a romper algo.

🔀

merge

Unir tu trabajo a main cuando todo está revisado y aprobado.

Cómo nombrar las ramas

En KaledSoft seguimos una convención: tipo/descripcion-corta

🚀 feature/
feature/kaledpark-qr-pago

Para agregar algo nuevo. Una feature que no existía antes.

Ej: nuevo módulo de pagos en KaledPark
🐛 fix/
fix/kaledental-citas-duplicadas

Para corregir algo que está roto o que no funciona bien.

Ej: las citas se estaban guardando dos veces
🔧 chore/
chore/actualizar-prisma

Para tareas de mantenimiento. No cambia lo que el usuario ve.

Ej: actualizar versión de la librería de base de datos
🚨 hotfix/
hotfix/kaledwash-pago-critico

Para urgencias en producción. Requiere atención inmediata.

Ej: los pagos en KaledWash están fallando ahora mismo

💡 Los nombres son como carpetas — te dicen de qué trata el trabajo antes de abrirlo

El Pull Request: pedir permiso para entrar

Un PR es una propuesta formal: "Aquí está mi trabajo, ¿lo revisamos antes de unirlo a main?"

Merge Conflict: cuando dos caminos chocan

No es un error del sistema. Es Git diciéndote: "Dos personas cambiaron la misma línea. ¿Cuál de las dos es la correcta?"

📄 src/kaledwash/precio.ts ⚠ CONFLICTO
1 export function calcularPrecio() {
2 <<<<<<< HEAD (tu rama)
3   return precioBase * 1.19; // IVA 19%
4 =======
5   return precioBase * 1.15; // IVA 15%
6 >>>>>>> main
7 }

¿Qué significa esto?

🟢

Tu rama dice que el IVA es 19%

🔴

main tiene el IVA en 15%

🤔

Git no puede adivinar cuál es correcto — tú debes decidir

💡 No entres en pánico

Un merge conflict no borra nada. Es solo Git poniéndote una pregunta. Hablas con tu compañero/a, acuerdan cuál es el correcto, eliminan los marcadores y listo.

✅ Cómo resolver un conflict en 4 pasos

1
Abre el archivo en VS Code
2
Decide cuál versión es la correcta
3
Borra los marcadores <<< ===== >>>
4
git add . && git commit

El flujo completo de KaledSoft

Así trabaja el equipo en cada feature, desde la idea hasta producción

📋

Crear la rama desde main

Siempre empiezas desde la versión más actualizada del código

git checkout main && git pull && git checkout -b feature/mi-feature
💻

Trabajar y hacer commits

Escribe tu código, haz commits frecuentes con mensajes descriptivos

git add . && git commit -m "feat: descripcion clara"
⬆️

Subir la rama a GitHub

Empuja tus cambios al repositorio remoto para que el equipo los vea

git push origin feature/mi-feature
🔍

Abrir un Pull Request

En GitHub, creas el PR y asignas un reviewer. La conversación técnica empieza aquí

GitHub → New Pull Request → base: main ← compare: feature/mi-feature
✅

Review → Aprobación → Merge

Una vez aprobado, el código entra a main y se despliega automáticamente a producción

Vercel detecta el push → deploy automático → ¡en producción!
⭐

La Regla de Oro de KaledSoft

Nunca hacemos commit directo a main.
Siempre trabajamos en ramas, siempre abrimos un Pull Request.
🚀 main → Vercel → producción real
🦷 50 clientes dependen de esto
🛡️ El PR protege a todos

Un bug en main a las 3 PM puede interrumpir el negocio de un odontólogo que tiene pacientes esperando, de un lavadero con carros en fila, o de un parqueadero lleno. Las ramas son la red de seguridad que nos protege a todos.

Resumen

Lo que aprendiste hoy sobre trabajo en equipo con Git

🌿

Ramas

Copias paralelas del código. Tu espacio seguro para trabajar sin afectar a main ni a tus compañeros.

🔀

checkout / merge

Moverse entre ramas y unir el trabajo terminado al código principal de forma ordenada.

🔍

Pull Request

Una propuesta de cambio con revisión humana. Ningún código llega a main sin que alguien más lo haya visto.

💥

Merge Conflict

Dos cambios en la misma línea. Git te pide que decidas. No es un error — es el sistema protegiéndote.

📋

Convención de nombres

feature/, fix/, chore/, hotfix/ — nombres que comunican el propósito de la rama antes de abrirla.

⭐

Regla de oro

Nunca commit directo a main. Siempre rama. Siempre PR. Siempre revisión.
