La historia de cómo se hablan
las computadoras

Desde el telégrafo hasta los MCP de inteligencia artificial — la necesidad de que las máquinas se comuniquen ha existido siempre.

1960s — ARPANET
🔬 Los primeros nodos conectados
El Departamento de Defensa de EE.UU. conecta 4 universidades. La primera transmisión fue "LO" (querían enviar "LOGIN" pero el sistema colapsó). La idea: una red descentralizada que sobreviva a un ataque nuclear.
1974 — TCP/IP
📦 Vint Cerf & Bob Kahn inventan TCP/IP
El protocolo que todo lo cambió. TCP divide la información en paquetes, los envía por la red y los reensambla al llegar. IP define la dirección de destino. Sin TCP/IP, no existe internet moderno.
1983 — DNS
🗺️ El directorio telefónico de internet
Antes de DNS había que memorizar IPs numéricas. DNS traduce "google.com" a "142.250.80.46". Es el sistema de nombres que hace internet usable para humanos.
1991 — HTTP/1.0
🌐 Tim Berners-Lee inventa la Web
Trabajando en el CERN, Berners-Lee crea HTTP, HTML y URLs. Su objetivo: que los físicos puedan compartir documentos. Accidentally inventa la World Wide Web. Una conexión por request.
1997 — HTTP/1.1
⚡ Conexiones persistentes
Se mantiene la conexión abierta para múltiples requests. Cache control, virtual hosting. El web se vuelve más rápido y las aplicaciones dinámicas se vuelven posibles.
2000 — REST
🏗️ Roy Fielding define REST en su tesis doctoral
Arquitectura sin estado para construir APIs escalables sobre HTTP. Se convierte en el estándar de facto para APIs web. Lo estudiaremos a fondo hoy.
2015 — HTTP/2
🚀 Multiplexing y compresión
Múltiples requests simultáneos sobre una sola conexión. Compresión de headers. Servidor push. La web moderna necesitaba esto para cargar decenas de recursos en paralelo.
2022 — HTTP/3 & QUIC
💨 UDP reemplaza a TCP
HTTP/3 corre sobre QUIC (basado en UDP). Conexiones más rápidas, sin "head-of-line blocking". TikTok, YouTube y Google ya lo usan. 26% del tráfico web es HTTP/3.
2024 — MCP (Model Context Protocol)
🤖 Anthropic define el estándar para conectar IAs
La misma necesidad de 1974: que los sistemas se comuniquen. Ahora los agentes de IA necesitan su propio "TCP/IP". MCP es el protocolo que permite a Claude, GPT y otros modelos conectarse a herramientas externas de forma estándar.
TCP/IP — la magia de los paquetes
Enviando REQUEST del cliente al servidor...
Iniciando transmisión...
TCP divide tu mensaje en paquetes numerados, los envía por cualquier ruta disponible, y los reensambla en destino en el orden correcto. Si un paquete se pierde, se reenvía automáticamente.

Interfaces de programación

¿Qué es una API?

Application Programming Interface — una interfaz que permite que dos sistemas se comuniquen sin que ninguno necesite conocer la implementación interna del otro.

💡 Una API no es solo para navegar internet. Es cualquier contrato de comunicación entre sistemas. Cuando creas un dispositivo electrónico nuevo (un sensor, un parlante bluetooth, un POS), defines una API para que otros lo puedan usar.

Arquitectura de software

REST — Representational State Transfer

🎓
2000 — UC Irvine, California

Roy Fielding

En su disertación doctoral "Architectural Styles and the Design of Network-based Software Architectures" define REST. Era uno de los autores originales de HTTP/1.1. REST no es un estándar, es un estilo arquitectónico.

REST describe cómo usar HTTP correctamente para construir sistemas escalables, simples y predecibles.
Los 6 principios de REST
🚫🧠
Sin Estado (Stateless)
Cada request tiene todo lo necesario. El servidor no guarda sesiones.
🔗
Interfaz Uniforme
URLs para recursos, verbos HTTP para acciones.
💾
Cacheable
Las respuestas indican si pueden guardarse en caché.
🏗️
Sistema en Capas
Cliente no sabe si habla directo o por un proxy/balanceador.
🎭
Client-Server
UI y lógica separadas. Frontend y backend independientes.
📦
Code on Demand (opcional)
Servidor puede enviar código ejecutable (JavaScript).
¿Por qué separar frontend y backend?
🔄

Un backend, N frontends

La misma API de KaledDental sirve la app iOS, Android, web y el dashboard del odontólogo. Sin duplicar lógica.

🚀

Deploy independiente

Actualizas el backend sin tocar el frontend. Puedes escalar solo el backend cuando hay pico de citas.

👥

Equipos paralelos

El team de iOS y el team de backend trabajan en paralelo, coordinándose solo en el contrato de la API.

Metadatos de cada request

    
HTTP Headers

    
Los headers son pares clave-valor que viajan con cada request y response. Contienen instrucciones, credenciales, formatos y contexto. Un senior developer lee headers instintivamente.


    
headers de request más importantes

    

      

        Authorization: Bearer eyJhbGci...
        JWT token de autenticación
        👤 Quién eres
      

      

        Content-Type: application/json
        Formato del body que envías
        📄 Qué formato mandas
      

      

        Accept: application/json
        Formato que aceptas en respuesta
        📥 Qué formato quieres recibir
      

      

        Cache-Control: no-store
        No guardes esta respuesta en caché
        🔒 Crítico para datos médicos
      

      

        X-Request-Id: abc-123-xyz
        ID único para rastrear el request
        🔍 Trazabilidad en logs
      

      

        Origin: https://app.kaleddental.com
        De dónde viene el request (CORS)
        🌐 Seguridad cross-origin
      

    


    
headers de response más importantes

    

      

        Content-Type: application/json
        Formato del body que te devuelvo
        📤 Qué formato recibes
      

      

        Cache-Control: max-age=3600
        Puedes cachear esto por 1 hora
        ⚡ Optimización de velocidad
      

      

        Access-Control-Allow-Origin: *
        Cualquier dominio puede usar esta API
        🌍 CORS policy
      

      

        Retry-After: 60
        Vuelve a intentar en 60 segundos
        ⏱️ Rate limiting feedback
      

      

        Set-Cookie: session=abc; HttpOnly
        Guarda esta cookie (solo HTTP, no JS)
        🍪 Sesión segura

Propiedad matemática en HTTP

    
Idempotencia

    
Una operación es idempotente si ejecutarla múltiples veces produce el mismo resultado que ejecutarla una sola vez. Concepto clave para sistemas de pago y operaciones críticas.


    

      Definición formal: f(f(x)) = f(x). En HTTP: hacer el request N veces tiene el mismo efecto en el sistema que hacerlo 1 vez.
    


    

      

        
MÉTODO

        
DESCRIPCIÓN

        
IDEMPOTENTE

      

      

        GET
        Leer datos del servidor. No modifica nada. Puedes llamarlo 100 veces, el estado no cambia.
        ✓
      

      

        PUT
        Reemplaza el recurso completo. Llamarlo 5 veces con los mismos datos = mismo resultado que 1 vez.
        ✓
      

      

        DELETE
        Elimina el recurso. Llamarlo una segunda vez retorna 404, pero el estado del sistema es el mismo.
        ✓
      

      

        PATCH
        Depende de la implementación. Puede o no ser idempotente según cómo definas la lógica.
        ~
      

      

        POST
        Crear recurso. Llamarlo 2 veces crea 2 recursos. NO es idempotente.
        ✗
      

    


    

      

        💳
        
El problema del doble cobro

        
Usuario paga en KaledPark. La app tiene error de red y reintenta. Si POST no tiene control de duplicados → cobras dos veces. Esto destruye la confianza del usuario.

      

      

        🔑
        
Idempotency Key

        
Solución: enviar un UUID único en el header Idempotency-Key. Si el servidor ya procesó ese key, retorna la respuesta original sin cobrar de nuevo. Stripe lo usa así.

        

POST /api/pagos
Idempotency-Key: pay-uuid-abc-123

reglas de oro del diseño

    

      

        📌
        
Versionado

        
Siempre /api/v1/. Cuando hagas cambios breaking, crea /api/v2/ sin romper los clientes actuales.

      

      

        🔍
        
Filtros y Paginación

        
?page=2&per_page=20&status=activo. Nunca retornes 10,000 registros sin paginar.

      

      

        🔒
        
Seguridad por defecto

        
HTTPS siempre. JWT en headers, no en query params. Rate limiting. Validar y sanitizar todos los inputs.
